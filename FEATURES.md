# FamilyShield — Product & Feature Overview

FamilyShield is a **parental-safety app** that lets a parent keep their children safe and
connected: see where each child is in real time, review where they've been, get alerted
when something needs attention, chat with them, and understand how they spend time on
their phone — all from one dashboard.

It is built around **"Digital Guardianship"**: the authority and reliability of a security
product with the warmth of a family tool. The design language is Material 3, a white +
deep-navy / sky-blue palette, the Hanken Grotesk typeface, and **open-source maps only**
(OpenStreetMap / osmdroid — never Google Maps).

---

## The two apps

FamilyShield ships as **one Android app** with two experiences plus a shared backend:

| Experience | Who | Purpose |
|---|---|---|
| **Parent** | The guardian | Sign in, manage children, see locations/history/usage, set safe zones, chat, change settings. The app opens here by default. |
| **Kid device** | The child's phone | A pairing-code flow (no login) that links the device to a parent, then reports location / battery / app usage and lets the child chat back. (In this repo the kid side is a **simulator** for testing.) |
| **Backend** | — | A REST API + PostgreSQL/PostGIS data core the apps consume. |

## Features

### 1. Accounts & authentication
- Parent **register / sign in** with email + password.
- Passwords hashed with **argon2id**; sessions use **JWT** (15-minute access token, 30-day
  refresh token) with transparent auto-refresh in the app.
- Login is **rate-limited** to slow brute-force attempts; errors are generic (no account
  enumeration). The app starts on the parent experience.

### 2. Children & device pairing
- Add, rename, and manage **multiple children**.
- Pair a child's phone with a **6-digit pairing code** (generated by the parent, entered on
  the kid device). The kid app needs **no account** — just the code.
- Each paired device reports platform, model, battery, charging state, and last-seen time.

### 3. Real-time location tracking
- A **live map** of where each child is, refreshed automatically (polling) while the app is
  open — no manual refresh.
- The **Family Network** dashboard shows **all children at once** on one map with labelled
  markers, plus a "Full View" full-screen map.
- Maps are **OpenStreetMap** via osmdroid; reverse-geocoding (turning coordinates into place
  names) uses OSM Nominatim. No Google services.

### 4. Location history & route detection
- A **per-day timeline** of recorded locations for each child.
- **Route detection** finds where a child **stayed** (stops), the **trips** between them
  (departure → return points), and **frequently-recurring routes** (e.g. home ↔ school),
  shown with distance, duration, and a mini-map.
- **Top Places Today** on the dashboard surfaces the places a child visited today, with
  reverse-geocoded names and time ranges.

### 5. Safe zones (geofencing)
- Define **safe zones** (name, center, radius) per child — e.g. Home, School.
- Zones are drawn on the map; the app indicates when a child is **currently inside** a zone.
- Inside a chat thread, a **Safe-Zone card** appears when the child is currently in one of
  their zones.

### 6. Alerts
- **Low-battery** and **offline** alerts per child, surfaced as **named Recent Alerts**
  ("Noah: Low battery") on the dashboard.
- Backend supports push delivery via **FCM** (firebase-admin) when configured; an
  offline-sweep cron endpoint flags devices that stop reporting.

### 7. Multi-child "Family Network" dashboard
- **Child-filter chips** ("All Children" + one per child) to scope the whole dashboard.
- **Active-monitoring hero** with the multi-child map and online count.
- **Top Places Today**, **Recent Alerts**, and **Quick Actions** (Ring device, Timeline,
  Emergency), plus a notifications bell and settings access.

### 8. Chat (WhatsApp-style) — parent ⇄ kid
- Per-child text chat with **conversation list** (avatar, last message, unread badge) →
  **thread** with chat bubbles, timestamps, and **read receipts**.
- Header shows the child's **online status**; a **"Today" date chip**; **quick-reply chips**
  (parent: *Love you!* / *Stay safe!* / *Call me later?*, kid: *I'm safe!* / *On my way!* /
  *Pick me up?*); and a **location-request** button in the input bar.
- Near-real-time via 3-second polling (consistent with live tracking). The kid device has
  its own chat screen.

### 9. App Usage (screen time)
- Per-child **screen-time** view: **total today** (with % change vs yesterday), a **weekly
  trend** bar chart with today highlighted and the weekly average, and a **per-app
  breakdown** (app icon + category + duration + relative usage bar) for YouTube, Roblox,
  WhatsApp, TikTok, Chrome, Spotify, Instagram, Minecraft, etc.
- An **App Limits** entry point (placeholder for daily time allocations).
- Data model + API are real; on a real device the kid app would source per-app minutes from
  Android's `UsageStatsManager` (requires the `PACKAGE_USAGE_STATS` permission). In this
  repo the data is seeded for the demo.

### 10. Settings
- **Manage profiles** (add child, rename, per-child status).
- **Privacy Lock** (biometric-gate toggle), **Zones** and **Alerts** shortcuts.
- **Language** picker (see below) and **Log out**.

### 11. Localization & RTL
- Full **English** and **Hebrew** translations across both apps; selecting Hebrew flips the
  entire UI to **right-to-left** automatically.
- **In-app language toggle** in Settings (System default / English / עברית), persisted and
  applied immediately. Defaults to the phone's language.

### 12. Security
- Secrets **fail closed** — the app refuses to mint/verify tokens if `JWT_SECRET` /
  `JWT_REFRESH_SECRET` are unset (no predictable-key fallback). JWT verification is pinned
  to **HS256**.
- Every child-scoped API route enforces **ownership** (a parent can only access their own
  children — no IDOR). Cron endpoint uses a constant-time secret check.
- Android: **`allowBackup="false"`** (tokens can't be pulled via `adb backup`); **cleartext
  HTTP is debug-only** (release builds are HTTPS-only).

## Architecture & tech stack

```
┌──────────────────────────┐        HTTPS/JSON        ┌───────────────────────────┐
│  Android app (Kotlin)    │ ───────────────────────► │  Backend (Next.js API)    │
│  • Parent experience     │                          │  • REST routes (App Router)│
│  • Kid-device experience │ ◄─────────────────────── │  • Drizzle ORM             │
│  Jetpack Compose · M3     │     tokens / data        │  • Zod validation          │
│  osmdroid (OSM maps)     │                          │  • jose JWT · argon2       │
└──────────────────────────┘                          └─────────────┬─────────────┘
                                                                     │
                                                       ┌─────────────▼─────────────┐
                                                       │ PostgreSQL + PostGIS      │
                                                       └───────────────────────────┘
```

**Backend** — Next.js 16 (App Router, Node runtime) · TypeScript · Drizzle ORM + `pg` ·
PostgreSQL + **PostGIS** · Zod · `jose` (JWT) · `argon2` · `firebase-admin` (FCM) ·
OpenAPI (`zod-to-openapi`) + Swagger UI · Vitest. Containerized with Docker Compose.

**Android** — Kotlin · Jetpack Compose · **Material 3** (+ Material 3 Adaptive) · Navigation
Compose · OkHttp + kotlinx.serialization · **osmdroid** (OpenStreetMap) · Hanken Grotesk.
ViewModels are dependency-injected (`ApiClient` / `TokenStore` interfaces) for JVM unit
testing with hand-written fakes.

### Data model (core tables)
`parents` · `children` · `devices` · `pairing_codes` · `locations` (PostGIS points,
partitioned) · `safe_zones` · `alerts` · `messages` (parent⇄kid chat) · `app_usage`
(per-app daily screen time).

### API surface (selected)
```
POST   /api/auth/register | login | refresh
GET    /api/children                       POST /api/children
GET    /api/children/[id]/location/current
GET    /api/children/[id]/location/history
GET    /api/children/[id]/routes
GET    /api/children/[id]/alerts
GET    /api/children/[id]/zones            POST /api/children/[id]/zones
GET    /api/children/[id]/messages         POST /api/children/[id]/messages
GET    /api/children/[id]/app-usage
POST   /api/children/[id]/pairing-code
POST   /api/pair                           (kid device pairs with a code)
POST   /api/locations | /api/device/status | /api/device/messages | /api/device/app-usage
GET    /api/docs  ·  /api/openapi.json     (Swagger UI + OpenAPI document)
```
Parent routes require a Bearer **access token**; kid-device routes require a **device token**.

## Running it

### Backend (Docker)
```bash
docker compose up -d        # PostGIS db + backend on http://localhost:3000
```
The backend reads `JWT_SECRET`, `JWT_REFRESH_SECRET`, `CRON_SECRET`, `DATABASE_URL`, etc.
from the environment (see `.env.example`). **Set strong unique secrets for production.**

### Backend (local dev)
```bash
npm install
npm run dev                 # Next.js dev server
npm test                    # Vitest suite
```

### Android app
```bash
cd android
# Emulator build (reaches the host backend at 10.0.2.2):
./gradlew :app:assembleDebug
# Physical device build — point at your PC's LAN IP:
./gradlew :app:assembleDebug -PAPI_HOST=192.168.x.x
```
Install the APK, open the app, and sign in. For a physical phone the device must be on the
same Wi-Fi and the backend reachable (open the firewall for port 3000).

### Demo data & test accounts
Seed scripts populate realistic data against a running backend / database:
```bash
node scripts/seed-demo.mjs          # parent + 4 kids w/ locations, zones, history, routes
node scripts/seed-app-usage.mjs     # 7 days of per-app screen time for every child
```
